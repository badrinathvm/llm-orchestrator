PLAN

Implement the "Adaptive Routing" LLM orchestration pattern

Context

The reference image /Users/ranibadri/Documents/projects/Cowork/llm_orchestration_enterprises.png shows 6 enterprise LLM orchestration patterns. The user wants to implement the Routing (Adaptive) pattern:

Query → Router → { Small model | Frontier model | Specialist } → Reply
"Best for: Support triage, cost optimisation"

The router inspects the incoming query and dispatches it to the cheapest model capable of handling it (a fast/cheap "small" model for simple queries, a powerful "frontier" model for complex reasoning, or a "specialist" model/prompt for domain-specific queries), instead of always paying for the most expensive model.

The project already has a minimal LangGraph scaffold from the previous session (graph/graph_builder.py, nodes/nodes.py, state/orchestration_state.py, llm/llm.py, server.py, models/models.py) with a single placeholder generate node wired START → generate → END. This task replaces that placeholder graph with a real routing graph, reusing the existing OpenAILLM wrapper, OrchestrationState, GraphBuilder, and FastAPI plumbing.

Design

Router: an LLM call with structured output that classifies the query into one of three routes — small, frontier, specialist — plus a short reason (useful for the support-triage/cost-optimization story: you can log why a query was routed where).

Responders (one node per branch, all returning {"result": ...}):

  • small_model_responder — cheap/fast model (gpt-4o-mini) for simple/short queries (e.g. FAQ-style, greetings)
  • frontier_model_responder — most capable model (gpt-4o) for complex/multi-step reasoning
  • specialist_responder — gpt-4o with a domain-specific system prompt (e.g. a support/triage specialist persona) for domain-specific queries

Graph wiring (GraphBuilder.build_graph):

START → router →(conditional on state["route"])→ { small_model_responder | frontier_model_responder | specialist_responder } → END

This mirrors the conditional-edge pattern already used in the KnowledgeGraph reference (KnowledgeGraph/graph/graph_builder.py:16-20, routing to rerank vs generate based on state["strategy"]), just with three branches instead of two.

Files to change

  1. llm/llm.py — generalize OpenAILLM to accept a model (and optional temperature) so different routes can use different models:
    • __init__(self, model: str = "gpt-4o", temperature: float = 0.0), store and pass through to ChatOpenAI(api_key=..., model=self.model, temperature=self.temperature).
    • Keeps the existing try/except + load_dotenv() structure; just parametrizes the hardcoded "gpt-4o".
  2. state/orchestration_state.py — add routing metadata fields:
    • route: str ("small" | "frontier" | "specialist")
    • route_reason: str (router's justification, surfaced for triage/cost-optimization visibility)
  3. prompts/__init__.py (empty) and prompts/prompt.py (new package, mirroring KnowledgeGraph/prompts/prompt.py):
    • A router classification prompt instructing the LLM to pick one of small/frontier/specialist based on query complexity/domain, with a brief reason.
    • A specialist system prompt (e.g. a "senior support triage specialist" persona) used by specialist_responder.
  4. nodes/nodes.py — replace the placeholder generate with:
    • route_query(state) -> dict: builds an LLM via OpenAILLM().get_llm(), calls .with_structured_output(...) against a small Pydantic schema (route: Literal["small","frontier","specialist"], reason: str) using the router prompt from prompts/prompt.py, returns {"route": ..., "route_reason": ...}.
    • small_model_responder(state) -> dict: OpenAILLM(model="gpt-4o-mini").get_llm(), invoke directly with the query, return {"result": response.content}.
    • frontier_model_responder(state) -> dict: OpenAILLM(model="gpt-4o").get_llm(), invoke directly with the query, return {"result": response.content}.
    • specialist_responder(state) -> dict: OpenAILLM(model="gpt-4o").get_llm(), invoke with the specialist system prompt + query (e.g. via ChatPromptTemplate or a simple message list), return {"result": response.content}.
  5. graph/graph_builder.py — rewire build_graph:
    • Add nodes router, small_model_responder, frontier_model_responder, specialist_responder.
    • add_edge(START, "router")
    • add_conditional_edges("router", lambda state: state["route"], {"small": "small_model_responder", "frontier": "frontier_model_responder", "specialist": "specialist_responder"})
    • add_edge(<each responder>, END)
    • Update run(query) to seed route and route_reason as "" in the initial state alongside existing query/context/result.
  6. models/models.py — extend QueryResponse with route: str and route_reason: str so the API surfaces routing decisions (directly supports the "support triage / cost optimisation" use case — callers can see which model tier handled their request and why).
  7. server.py — pass the new fields through: QueryResponse(answer=response["result"], route=response["route"], route_reason=response["route_reason"]).

Verification

  • python -c "from graph.graph_builder import GraphBuilder" and other modules still import cleanly (no circular imports between nodes, prompts, state, llm, graph).
  • Start the server (uv run python server.py or the existing "Debug API Server" VS Code launch config) and exercise /query with a few different kinds of questions, e.g.:
    • A trivial greeting ("hi", "what's 2+2") → expect route: "small"
    • A complex multi-step reasoning question → expect route: "frontier"
    • A support/triage-flavored question → expect route: "specialist"
    Confirm the route and route_reason fields in the JSON response plausibly match the query, and that answer is non-empty (requires a valid OPENAI_API_KEY in .env).
  • Optionally add a log line in route_query (or rely on DEBUG/uvicorn --log-level debug) to observe the router's decision per request while testing.
