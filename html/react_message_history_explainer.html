How the agent decides what to call next

There's no planner deciding "call this tool next." reason() re-reads the entire message history on every pass, and the LLM decides fresh each time — because every prior observation is sitting right there in the transcript it's about to read again.

Two separate lists, one job each

ReactState.messages uses add_messages as its reducer, so it only ever grows — nothing is edited or removed once appended. Every reason() call passes this whole list to the LLM, which is the entire mechanism: the model isn't told what to call next by any code in this repo, it infers it by reading what it already asked for and what came back, same as you would scrolling back up a chat.

A tool_call_id is the one piece of bookkeeping that matters here — it's how a ToolMessage gets matched back to the specific call that requested it, which is required once more than one call lands in the same turn, as happened at [1] above.

messages
